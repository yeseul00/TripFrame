## 2. 포지셔닝 분석

### 2.1 차별점: Pre-departure 특화

TripFrame만의 기능 2축:
1. **역산 엔진**: 항공편 출발 시간으로부터 집 출발 시간을 역방향 계산
2. **공백 감지**: 예약 이벤트 사이 이동수단 누락 자동 발견

→ 두 기능 모두 어떤 주요 여행 앱에서도 제공하지 않음

### 2.2 포지셔닝 유효성

| 평가 항목 | 판정 | 근거 |
|-----------|------|------|
| 니치 존재 여부 | ✅ 유효 | 이동 로지스틱 특화 앱 부재 |
| 경쟁 앱과 비겹침 | ✅ 확인 | 트리플/마이리얼트립과 보완 관계 |
| 독립 생존 가능성 | ⚠️ 주의 | 사용 빈도 한계, 기능 흡수 위험 |

### 2.3 전략 방향: 듀얼 트랙

```
Track A (B2C 소비자 앱) — Phase 4~6, 지금~6개월
├── D-day 위젯 (항상 켜진 접점)
├── Affiliate 교통 예약
├── 보험 연계 수수료
├── Pro 구독
└── Import/Export 데이터 브리지

Track B (B2B 엔진 API) — Phase 7+, 6개월~1년
├── OTA 통합 (역산 API)
├── 여행사 화이트라벨
├── 기업 출장 도구
└── 교통 데이터 라이선싱

두 트랙 공유: @tripframe/core (플랫폼 독립 순수 TS 엔진)
```

## 3. 사용 빈도 한계 & D-day 위젯 전략

### 3.1 문제

여행은 연 1~3회 이벤트. 앱 사용이 여행 직전 3~5일에 집중 → DAU/MAU 극도 저하 위험.

### 3.2 해결: D-day 위젯

| 지표 | 위젯 없음 | 위젯 있음 |
|------|----------|----------|
| 앱 접촉 빈도 | 여행 전 3~5일 | 예약 직후~출발일 매일 |
| 잠금화면 노출 | 0회/일 | 1~10회/일 |
| 감정 연결 | 도구적 | 정서적 (기대감) |
| 재방문 트리거 | 앱 이름 기억 못함 | 위젯 탭 → 즉시 진입 |

**차별화**: 단순 D-day 카운트가 아닌 **행동 유도 알림** 포함
- "D-7: 유후인 버스 예약 오픈!"
- "D-3: 짐 크기 기준 택시 예약 추천"
- "D-1: 출발 체크리스트 확인"

### 3.3 구현 전략

- iOS: WidgetKit + React Native Bridge (expo-widgets)
- Android: App Widget
- Web: PWA 알림
- 시점: Phase 5 (Development Build 전환 후)

**판정**: 사업 생존 요건. "있으면 좋은 기능"이 아님.

## 4. 수익 모델: 3단 퍼널

### Layer 1: Affiliate (교통 예약 수수료)

- 현재 상태: `TransportOption.bookingUrl` 필드 + `WebBrowser.openBrowserAsync()` 이미 구현
- 필요 작업: affiliate 파라미터 추가만
- 예상 단가: 교통 예약 건당 1,000~1,500원 (요금의 3~5%)

### Layer 2: Insurance Referral (여행 보험 연계)

TripFrame이 이미 보유한 데이터(여행 기간, 목적지, 항공편, 인원수) = 보험 견적 핵심 입력값.

| 연계 대상 | 접근 방식 | 예상 수수료 |
|---------|---------|-----------|
| 삼성화재 다이렉트 | API/딥링크 제휴 | 보험료의 15~25% |
| 메리츠화재 | 비교 플랫폼 경유 | 10~20% |
| 보험 비교 플랫폼 (굿리치 등) | CPA | 리드당 3,000~5,000원 |

구현 위치: 역산 탭 "D-7" 시점 제안 카드 또는 D-day 위젯 체크 항목.

### Layer 3: Freemium (Pro 구독)

| 무료 | Pro (월 2,900원 / 여행당 3,900원) |
|------|------|
| 1개 여행 관리 | 무제한 여행 |
| 기본 역산 (1개 교통수단) | 대안 교통수단 비교 |
| 수동 입력만 | 이미지/메일 자동 입력 |
| 로컬 저장 | 클라우드 동기화 |
| D-day 위젯 (기본) | D-day 위젯 + 행동 알림 |

## 5. 방어 가능성 (Moat)

### 위협: 대형 앱의 기능 흡수

역산 엔진 알고리즘 자체는 모방 용이 (단순 뺄셈 + 인접 비교). 트리플이 3개월 안에 비슷한 기능 구현 가능.

### 해자 3가지

**Moat 1: 교통 데이터 축적** (가장 강력)
- Phase 4에서 `TransportRoute` 구조화 DB 전환
- 사용자 크라우드소싱 (실제 이용 소요시간, 비용, 만족도)
- 시간이 지날수록 데이터 우위 → 모방 비용 기하급수 증가

**Moat 2: Import/Export 데이터 브리지**
- Export: iCal, CSV, Google Calendar → TripFrame이 "일정의 원천"
- Import: 예약 메일, SMS, 스크린샷에서 자동 일정 추출 → 진입 장벽 제거

| 입력 소스 | 기술 | 난이도 | Phase |
|---------|------|--------|-------|
| CSV/iCal 파일 | 표준 파서 | 하 | Phase 4 가능 |
| 예약 확인 이메일 | Gmail API + 파싱 | 중 | Phase 5+ |
| SMS | Android Permission + 파싱 | 중 | Phase 5+ |
| 스크린샷/이미지 | OCR / Vision API | 상 | Phase 6+ |

**Moat 3: 오프라인 우선 아키텍처**
- Constitution Article III-5 준수
- 비행기 안, 해외 로밍 제한 상황에서도 핵심 기능 동작
- 기존 슈퍼 앱의 서버 의존 아키텍처에서 전환 비용 높음

### Phase 4 즉시 반영 가능 항목

1. ✅ iCal/CSV Export — Core에 순수 함수 추가 (난이도 하)
2. ✅ 보험 제안 카드 UI — 제안카드 패턴 재사용 (난이도 하)
3. 📋 D-day 위젯 설계 문서 — Phase 5 선행 설계 착수

### 핵심 권고사항

1. D-day 위젯은 **사업 생존 요건** — backlog 우선순위 최상향
2. 독립 앱(Track A)과 엔진 API(Track B)를 **동시 추구** — @tripframe/core 분리가 이를 가능하게 함
3. 수익 모델 **보험 연계**를 Phase 4에 UI만이라도 선배치
4. **CSV/iCal Import**는 Phase 4에서 바로 착수 가능

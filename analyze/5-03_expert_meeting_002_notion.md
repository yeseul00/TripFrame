### A-1. EAS Dev Build 즉시 전환 (결정 1)

| 참석자 | 의견 |

| Expert 2 | **강력 동의**. Phase 4에서 Expo Go 한계가 이미 드러남 (expo-secure-store 불가). Dev Build 전환은 Phase 5의 게이트키퍼 — 이것 없이는 보안 업그레이드도 위젯도 불가. 다만 **첫 빌드 시 CocoaPods/Gradle 의존성 해결에 반나절~1일 소요** 예상. 태스크 추정에 버퍼 포함 필요. |

### A-2. expo-secure-store 마스터 키 업그레이드 (결정 2)

| 참석자 | 의견 |
|--------|------|
| Expert 2 | **동의 + 보완**. 마이그레이션 설계(kv-store → SecureStore 단방향)는 올바름. 추가 고려: **SecureStore 저장 실패 시 폴백 전략**. 기기 Keychain 용량 부족 또는 보안 정책으로 저장 실패하는 edge case 존재. 실패 시 kv-store 키를 삭제하지 않고 유지하는 방어 로직 필요. |
| Expert 3 | 사용자에게 투명한 전환이 핵심. "데이터가 날아갔다" 경험은 절대 불가. 마이그레이션 중 로딩 인디케이터 표시 권고. |
| Expert 1 | 보안 업그레이드는 마케팅에도 활용 가능 — "하드웨어 보안 모듈로 데이터 보호". 앱 스토어 설명에 포함 권고. |

**검증 결과**: ✅ 동의. 폴백 안전장치 보완 필요.
**🟡 누락 발견**: SecureStore 저장 실패 시 폴백 로직 미설계.

### A-3. 예약 루프 — resolvedGapIds 외부 관리 (결정 3)

| 참석자 | 의견 |
|--------|------|
| Expert 2 | **동의. Constitution 원칙 준수 관점에서 최선의 선택**. Gap은 `detectGaps(events)` 호출마다 파생 계산되므로, Gap 객체 내부에 상태를 넣으면 순수 함수 원칙이 깨짐. `resolvedGaps: { [tripId]: { [gapKey]: { resolvedAt, method } } }` 구조 적절. 다만 **gapKey 안정성**에 대한 단위 테스트 필요 — 이벤트 시간이 미세 변경되면 gapKey가 달라질 수 있음. |
| Expert 3 | UI 플로우에 대한 보완 의견: "예약 완료?" 다이얼로그가 **앱 복귀 시 자동 팝업**되면 사용자가 당혹할 수 있음. 대신 Gap 카드에 **"예약 완료 표시" 버튼**을 상시 노출하는 것이 더 자연스러움. AppState 감지는 보조 트리거로만 사용. |
| Expert 1 | RESOLVED 상태 데이터는 향후 **전환율 분석**에 활용 가능. "어떤 교통수단이 가장 많이 예약 완료 되는가" → affiliate 파트너 선정 근거. 분석을 위해 `method` 필드에 교통수단 유형 기록 권고. |

**검증 결과**: ✅ 동의. UI 트리거 방식 + gapKey 안정성 테스트 보완.
**🟡 누락 발견**: gapKey 생성 로직의 안정성 단위 테스트. AppState 자동 팝업 대신 상시 버튼 방식 권고.

### A-4. 4탭 구조 "이동 체크" (결정 4)

| 참석자 | 의견 |
|--------|------|
| Expert 3 | **동의. 4탭이 5탭보다 인지 부하 낮음**. "이동 체크"는 무난하나, 앱 고유 용어를 만드는 것도 고려할 가치 있음. 예: "루트 체크", "이동 보드". 다만 Phase 5에서 이름 실험(A/B)은 과잉이므로 "이동 체크"로 확정하고, 사용자 피드백 수집 후 Phase 7에서 재검토. |
| Expert 2 | 기술적 주의사항: 두 화면의 Zustand 상태 통합 시 **기존 E2E 97개 중 GapDetection + Suggestion 관련 테스트가 깨질 것**. 탭 재편 태스크에 E2E 수정 작업을 번들링해야 함. 별도 태스크로 분리하면 "테스트 통과하지 않는 시점"이 생김. |
| Expert 1 | "이동 체크" 이름 확정에 동의. 사용자에게 "이동수단 빠진 곳 확인 + 해결"이라는 행위를 한 단어로 전달해야 하는데, "이동 체크"가 가장 직관적. |

**검증 결과**: ✅ 만장일치 동의. E2E 수정 번들링 주의.
**🟡 누락 발견**: 탭 재편 태스크에 E2E 업데이트를 반드시 함께 포함해야 함 (분리 금지).

### A-5. Playwright 주력 + Maestro 보완 (결정 5)

| 참석자 | 의견 |
|--------|------|
| Expert 2 | **강력 동의. 원래 "Maestro 주력 전환" 권고가 과도했음을 인정**. 97개 Playwright 테스트는 Phase 1~4의 전체 비즈니스 로직을 검증하는 자산. 이걸 버리고 Maestro로 재작성하는 건 "테스트의 테스트를 만드는" 비효율. Maestro는 **SecureStore, AppState, Widget** 등 웹에서 불가능한 3~4개 시나리오에만 한정. |
| Expert 3 | 웹 확장 관점에서도 타당. Phase 2(Constitution Article II)에서 React 웹앱을 예정하고 있으므로, Playwright 테스트 기반이 웹앱에 거의 그대로 재사용 가능. |
| Expert 1 | 다만 **네이티브 전용 UX가 증가하면 Maestro 비중도 자연스럽게 올라갈 것**. Phase 6 D-day 위젯, Phase 7 푸시 알림 등. 현재 비율(Playwright 95% : Maestro 5%)이 Phase 8쯤엔 70:30 정도가 될 수 있음. |

**검증 결과**: ✅ 만장일치 동의. 비율은 네이티브 기능 증가에 따라 자연 조정.

### A-6. Android 우선 (결정 6)

| 참석자 | 의견 |
|--------|------|
| Expert 2 | **개발 환경 제약을 고려하면 유일한 합리적 선택**. iOS 기기 없이 WidgetKit 디버깅은 사실상 불가능 — EAS Cloud Build로 빌드는 되지만 위젯 레이아웃 확인은 Simulator/실기기 없이 blind flight. Android Studio + 에뮬레이터에서 즉시 확인 가능한 Android가 정답. |
| Expert 1 | **시장 데이터도 Android 우선 지지**. 한국 모바일 OS 점유율: Android 72%, iOS 28% (2025 StatCounter). 동남아 여행 타겟 확장 시에도 Android 우선이 유리. |
| Expert 3 | 동의하되 **iOS 사용자 경험이 열등해지지 않도록 주의**. 앱 자체(React Native)는 양쪽 동시 배포이므로, "Android 우선"은 **위젯·네이티브 확장 기능**에만 적용. 앱 본체의 iOS 품질은 동일하게 유지해야 함. |

**검증 결과**: ✅ 만장일치 동의. 앱 본체 iOS 품질 유지 확인.
**🟡 추가 합의**: "Android 우선"은 위젯·네이티브 확장에만 적용. 앱 본체는 iOS/Android 동시 배포 유지.

### A-7. iCal Export만 (결정 7)

|--------|------|
| Expert 1 | 동의. 다만 iCal Export 시 **TripFrame 고유 정보(Gap 상태, 역산 결과)를 어떻게 포함할지** 설계 필요. 표준 VEVENT만으로는 공백 감지 정보가 유실됨. `X-TRIPFRAME-GAP-STATUS` 같은 커스텀 프로퍼티 활용 가능. |

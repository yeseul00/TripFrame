## 2. 원인 상세 분석

### P1 — suggestion.spec.ts beforeEach 역효과

`MoveCheckScreen.tsx` 는 진입 시 첫 DANGER Gap을 **자동 펼침** 처리:
```ts
// MoveCheckScreen.tsx:217-223
const firstDanger = gaps.find((g) => g.severity === 'DANGER');
if (!openGapKey && firstDanger) {
  setExpandedIds(new Set([firstDanger.id]));
}
```

`suggestion.spec.ts` beforeEach가 이 상태를 고려하지 않고 다시 클릭:
```ts
// suggestion.spec.ts:19 — 이미 열린 카드를 닫아버림
await page.getByText('DANGER').first().locator('..').locator('..').click();
```

결과: 카드가 닫히고, SCR-004-02~08 전부 "교통 옵션 미표시"로 실패.

### P2 — TC-014 계열: 기본 펼침 상태 미고려

`gap.spec.ts:56`, `moveCheck.spec.ts:65` 테스트 의도:
> "Gap 카드를 탭하면 교통 옵션이 펼쳐진다"

실제 동작:
1. 진입 시 첫 DANGER Gap이 이미 펼쳐진 상태
2. 테스트가 클릭 → 닫힘
3. `expect(교통옵션).toBeVisible()` → FAIL

### P3 — ActionSheetIOS 웹 미지원

```ts
// HomeScreen.tsx:43-47 (현재)
if (Platform.OS === 'ios') {
  ActionSheetIOS.showActionSheetWithOptions(...)
} else {
  Alert.alert(...)  // Android
}
// 'web' 케이스 없음 → else(Alert)가 실행되나, 웹에서 Alert는 UI 없이 사라짐
```

웹에서 `Alert.alert()` 호출 → 아무 UI도 표시되지 않음 → CRUD-06~10 전부 실패.

### Phase 2 — TC-014 테스트 수정: 기본 펼침 상태 반영 (P2 해결)

**핵심 원칙**: 테스트는 "탭 → 펼침" 동작을 검증해야 함. 기본 펼침이므로, 먼저 닫은 뒤 열어서 검증.

#### 2-A. gap.spec.ts 수정

**수정 위치**: `apps/mobile/e2e/gap.spec.ts:56-65`

```ts
// 변경 전
test('[TC-014] DANGER Gap 카드 탭 시 교통 옵션 인라인 펼쳐짐', async ({ page }) => {
  await page.getByText('DANGER').first().locator('..').locator('..').click();
  await expect(page.getByText(/대중교통|택시|렌터카/).first()).toBeVisible();
});

// 변경 후 — 기본 펼침 상태이므로 닫은 뒤 다시 열어서 검증
test('[TC-014] DANGER Gap 카드 탭 시 교통 옵션 인라인 펼쳐짐', async ({ page }) => {
  const card = page.getByText('DANGER').first().locator('..').locator('..');
  await card.click();  // 닫기
  await expect(page.getByText(/대중교통|택시|렌터카/).first()).not.toBeVisible({ timeout: 3000 });
  await card.click();  // 다시 열기
  await expect(page.getByText(/대중교통|택시|렌터카/).first()).toBeVisible();
});

test('[TC-014-B] Gap 카드에 "예약 완료" 버튼 표시', async ({ page }) => {
  // 기본 펼침 상태이므로 바로 확인
  await expect(page.getByText('예약 완료').first()).toBeVisible();
});
```

#### 2-B. moveCheck.spec.ts 수정

**수정 위치**: `apps/mobile/e2e/moveCheck.spec.ts:65-100`

TC-014, TC-014-B~D, US-002-01~02 동일 패턴 적용:
- 기본 펼침 상태 확인 테스트 → 클릭 없이 바로 `expect(...).toBeVisible()`
- 토글 동작 테스트 → 닫기 후 열기 순서로 수정

**SCR-004-09, SCR-004-10 (설정 필터 테스트)** 별도 분석 필요:
- `설정` 탭에서 교통 선호 변경 후 `이동 체크` 탭에서 필터 결과 확인하는 테스트

**검증**: `npx playwright test apps/mobile/e2e/gap.spec.ts apps/mobile/e2e/moveCheck.spec.ts`

## 4. 작업 순서 및 의존성

```
Phase 0 (백업)
    ↓
Phase 1 (suggestion.spec.ts 수정)    ← 독립적, 가장 간단
    ↓
Phase 2 (gap / moveCheck spec 수정)  ← Phase 1과 독립
    ↓
Phase 3 (HomeScreen.tsx 수정)        ← Phase 1, 2와 독립
    ↓
Phase 4 (전체 재빌드 + Playwright)   ← 모든 수정 완료 후
```

Phase 1~3은 서로 독립적이므로 병렬 진행 가능.
Phase 4는 Phase 0~3 완료 후 반드시 순차 진행.
